Draft PR作成前のセルフレビューを実施してください。

## レビューの心得
- **まっさらな気持ちで**: 他人が書いたコードを初めて見る客観的な視点でレビューする
- **先入観を捨てる**: 「動いたから大丈夫」ではなく、本当に正しいか疑う

## 手順

### 1. 変更差分の確認
```bash
git diff main...HEAD
```

### 2. 設計原則チェック
- **DRY** (Don't Repeat Yourself): 同じロジックが複数箇所にコピーされていないか
- **KISS** (Keep It Simple, Stupid): 不必要に複雑な実装になっていないか
- **SOLID**: 各原則に違反していないか
- **YAGNI** (You Aren't Gonna Need It): 今必要でない機能を先回りで実装していないか

### 3. セキュリティレビュー
- **SQLインジェクション**: f-string SQL、プレースホルダ未使用（SQLAlchemy ORM/text()を使用すべき）
- **シークレット**: ハードコードされたAPIキー、パスワード、トークンがないか
- **XSS**: テンプレートでの未サニタイズ出力
- **情報漏洩**: エラーレスポンスで内部情報を露出していないか

### 4. コーディングレビュー
- snake_case（関数）、PascalCase（クラス）、型ヒント必須
- 不要な`print()`、デバッグコードが残っていないか
- async/await: 同期的なDB呼び出しが混入していないか

### 5. 過度な修正チェック（YAGNI再確認）
- 最小限の変更原則を守っているか
- 「ついでに」のリファクタリングをしていないか
- 使われない抽象化やヘルパーを作っていないか

### 6. 互換性
- 既存APIの後方互換性を壊していないか
- マイグレーションが必要な変更に対してalembicファイルが用意されているか
- 環境変数の追加がある場合、`.env.example`が更新されているか

### 7. テスト
- 新規コードに対するテストは十分か
- 既存テストがパスするか（`pytest -v --maxfail=5`）

（問題があれば詳細）
